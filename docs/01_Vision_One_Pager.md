# Vision 1pager — PoReSt (Portfolio + Notes + Blog + Resume)

## 1) 제품 비전 (한 문장)
공개 포트폴리오는 가볍게 보여주고, 오너 전용 공간에서 포트폴리오/이력서/지식노트/블로그를 “한 데이터”로 관리·재사용·연결하는 개인 개발자 OS.

## 2) 문제 정의
- 포트폴리오/이력서/노트/블로그가 분산되어 업데이트 비용이 높음
- 회사별 이력서 작성 시, 같은 경험을 반복 편집하며 품질이 흔들림
- 지식노트가 누적될수록 “연관 개념 탐색/정리” 비용이 급증
- 블로그 글은 표현/규칙 검수가 누락되어 품질 관리가 어렵고, 외부 플랫폼별 관리가 번거로움

## 3) 목표
### 핵심 목표
1) 공개 포트폴리오의 완성도/신뢰도 향상 (대표 프로젝트 중심, 전형적인 포트폴리오 형식)
2) 오너 전용 대시보드에서 경험/프로젝트/이력서 버전을 체계적으로 관리
3) 지식노트를 분야 무관하게 누적하고, 유사/연관 개념을 자동 추천 + 확정 연결로 그래프화
4) 블로그 작성 시 규칙 기반 표현 검출로 품질을 안정화하고, 외부 게시/관리는 “확장 가능한 방식”으로 설계

### 성공 지표 (초기)
- 포트폴리오: 대표 프로젝트 3개 이상 유지 + 프로젝트 상세 페이지 체류시간/스크롤 깊이 개선
- 이력서: 경험 원본 재사용률(한 경험이 2개 이상 버전에 재사용)
- 노트: 노트 100개 기준에서도 연관 개념 이동(탐색) 사용률 유지
- 블로그: 검출 항목 수정 후 재검수 통과율 증가

## 4) 사용자/권한 정책 (Public vs Private)
### Public (누구나)
- 메인 포트폴리오 홈 (/)
- 프로젝트 목록 (/projects)
- 프로젝트 상세 (/projects/[slug])
> 블로그/지식노트는 포트폴리오에 “콘텐츠로 섞지 않음”. 필요 시 메뉴 분기(링크)만 제공.

### Private (오너만: 로그인/인증 필수)
- 대시보드 홈 (/app)
- 포트폴리오 관리 (/app/portfolio) — 프로젝트/경험 CRUD, 대표 설정
- 이력서 관리 (/app/resumes) — 회사/직무별 버전, 경험 조합/정렬/스냅샷
- 지식노트 (/app/notes) — 노트 CRUD, 연관 후보 추천, 엣지 확정/해제
- 블로그 (/app/blog) — 글 CRUD, 표현 검출, Export/연동 상태 관리
- 피드백 (/app/feedback) — (후순위) 자동 피드백 실행/히스토리

## 5) 범위 (Scope)
### MVP 범위 (우선순위 반영)
1) 포트폴리오(공개) + 포트폴리오 관리(비공개)
2) 지식노트(비공개) + 연관 추천(후보) + 확정 연결(엣지)
3) 블로그(비공개) + 표현 검출(룰 기반) + Export
4) 피드백(후순위) — 우선 “룰/근거 정책 문서화 + 히스토리 저장”부터

### MVP 제외(비목표)
- 다중 사용자 SaaS
- 외부 블로그 플랫폼 전체에 대한 완전 자동 게시/수정(정책/제약으로 변동 리스크 큼)
- 노트 그래프의 고급 시각화(클러스터/중심성 등)는 2단계 이후

## 6) 핵심 설계 원칙
- “원본 데이터(경험/프로젝트/노트)”는 한 번만 작성하고, 이력서/뷰는 조합 결과로 만든다
- 자동화는 “후보 추천”과 “확정 연결”을 분리한다 (자동=제안, 확정=사용자 결정)
- 공개 영역은 최대한 가볍게(SSG/캐시), 개인 영역은 안전하게(서버 인증)
- 외부 연동은 “Connector 인터페이스”로 추상화하고, 초기에는 Export + URL/상태관리로 안정화

## 7) 기술 방향 (초기 권장)
- Web: Next.js(App Router) + TypeScript
- DB: PostgreSQL + Prisma
- 유사도/추천: pgvector(노트/키워드 임베딩 기반 연관 후보)
- Auth: 오너 전용(화이트리스트 이메일/매직링크/패스키 등 중 택1)
- Storage: 이미지/첨부는 S3 호환 스토리지
- 배포: Public(포트폴리오) 성능 최적화 + Private(대시보드) 인증 보호

## 8) 외부 블로그 연동 정책 (MVP)
- 1차: Export(HTML/Markdown 패키지 생성) + 외부 URL 등록/상태관리
- 2차: Connector 확장(지원 가능한 채널부터 단계적 추가)
- 이유: 플랫폼별 공식 API 정책/지원 범위 변화가 존재하므로, 서비스 핵심을 연동 리스크에 묶지 않는다
